Chapter 01

The problem RAG was built to solve.

15 minWhy retrieval exists
◈ Learning Objective

Understand why retrieval exists at all, and why vanilla LLMsLike a super-fast text helper that learned from a giant library. It guesses what words should come next. aren't enough.

The core insight

LLMs have two kinds of knowledge:

  • ParametricLike facts memorized for a test. They are already in the model's head from training. — baked into weightsLike tiny hidden knobs inside a machine. Those settings shape how the model answers. during training. Frozen, outdated, can hallucinate.
  • ContextualLike notes you hand someone right before they speak. The model uses that fresh info now. — whatever you put in the prompt. Fresh, accurate, limited by the context windowLike the amount of paper that fits on a desk at one time. Anything off the desk may get missed..

RAG = a system that dynamically fills the context window with the right parametric-like information at query time.

Key Concept

Two kinds of memory

Think of an LLM like a kid at school. What it learned before is in its head. The notes you give it now sit on the desk. RAG helps it use the right notes before it answers.

✧ Analogy

An LLM is a brilliant doctor with 10-year-old med school knowledge. RAG is handing them today's patient chart before they diagnose.

Plain English

AI knows many things, but some facts are old. RAG looks up fresh facts for each question and hands them to the AI before it speaks.

⟡ Exercise

Write down three questions your LLM cannot answer without retrieval. For example:

  • "What did our client file yesterday?"
  • "What's the current interest rate on BofA credit cards?"
  • "What's in the dispute letter I wrote last Tuesday?"
› Feynman Check
Why can't you just train the LLM on all your data?
Answer

Cost, latencyLike waiting for your food at a drive-through. It is the delay before the answer shows up., staleness, access control, and the fact that new data arrives constantly. Training a model takes days-to-months and freezes a snapshot of your data; retrieval takes milliseconds and stays fresh. You also can't easily apply per-user permissions to weightsLike tiny hidden knobs inside a machine. Those settings shape how the model answers., but you can apply them to a query.

△ Pitfall

Thinking RAG is about "giving the LLM more data." It's about giving it the right data at the right time.

Chapter 02

Regular RAG — the baseline.

20 minClassic pipeline & its limits
◈ Learning Objective

Build a mental model of the classic RAG pipeline and its limits.

The pipeline

Classic RAG: index once, query many. One-shot retrieval per question.

Key concepts

  • ChunkingLike cutting a long loaf into slices. Smaller pieces are easier to find and use.: slice docs into small pieces.
  • EmbeddingLike giving each piece of text a map pin based on meaning. Similar ideas land near each other.: turn text into numbers that capture meaning.
  • Vector similarityLike checking which songs feel alike in a playlist. It scores how close two meanings are.: how close two meanings are.
  • Top-kLike keeping only the top few search hits on page one. You keep the best handful to use.: the top N most similar results.
Key Concept

What an embedding actually is

An embedding is like a map of meaning. Sentences with close meaning land near each other, even if the words are not the same. That helps RAG find text like your question.

✧ Analogy

Think about how Spotify groups music. It doesn't sort songs by title — it sorts by how they actually sound. Chill acoustic tracks end up next to other chill acoustic tracks, miles away from death metal. When you search "rainy Sunday coffee shop," Spotify pulls songs that feel that way, even if none of their titles contain "rainy" or "Sunday." Vector search does the same thing with text: your question gets matched to paragraphs that mean the same thing, not ones that share the same words.

⟡ Exercise

Map out the pipeline for this question:

"What's our refund policy for enterprise customers?"

Walk through: chunking → embedding → retrieval → generation. Where could it go wrong?

› Feynman Check
Why does regular RAG fail on "Which clients in Texas owe over $10k?"
Answer

Vector similarity finds semantically similar text, not records matching filters. This is a structured query problem, not a semantic one. You need SQL (or a structured query language) to filter by state and balance. An embedding of the question looks similar to sentences about Texas clients and debt — it doesn't actually find which rows match those criteria.

△ The three fundamental limits of regular RAG
  1. One-shot: no re-querying if results are bad
  2. One source: only searches what you embedded
  3. No reasoning: can't combine sources or do multi-step lookups

This is exactly what agentic RAG fixes.

Chapter 03

The agentic leap.

20 minRetrieval as tool, not step
◈ Learning Objective

Understand what makes retrieval "agentic" and why it's a paradigm shift.

The shift

Regular RAG: retrieval is a step in a pipeline.
Agentic RAG: retrieval is a tool an agentLike a helper who can choose the next step on its own. It can use tools and keep working. chooses to use.

The agent loop (memorize this)

THINK → ACT → OBSERVE → repeat, until the agent decides to ANSWER.
  • THINK: the LLM reasons about what it needs
  • ACT: the LLM calls a tool (retrieval, calculator, API)
  • OBSERVE: the LLM sees the tool's result
  • LOOP: repeat until the LLM has enough to answer
Key Concept

Why "the loop" is the whole idea

Basic RAG may search one time and then answer. Agentic RAG can stop, check, search again, and keep going until the answer gets better.

✧ Analogy

Regular RAG = vending machine (one button, one snack).
Agentic RAG = personal chef (asks what you want, checks the fridge, checks the pantry, maybe runs to the store, cooks).

⟡ Exercise

Take this question: "Is our Q3 revenue higher than Q2, and what drove the change?"

Sketch the agent loop:

  1. Think: "I need Q3 revenue and Q2 revenue"
  2. Act: sql_query("SELECT SUM(amount) FROM sales WHERE quarter='Q3'")
  3. Observe: $2.4M
  4. Act: sql_query("SELECT SUM(amount) FROM sales WHERE quarter='Q2'")
  5. Observe: $2.1M
  6. Think: "Higher by $300k. What drove it?"
  7. Act: sql_query("SELECT product, SUM(amount) FROM sales WHERE quarter='Q3' GROUP BY product")
  8. ... and so on until answering.
› Feynman Check
What's the single structural difference between RAG and Agentic RAG?
Answer

The loop. Regular RAG is a straight line; agentic RAG is a cycle with a decision point at the top. Every other difference (multiple tools, re-querying, grounding checks) follows from the fact that the agent can decide to keep going.

Deep dive: tools

Tools are described to the agent via schemasLike a form with labeled boxes to fill in. It tells the AI exactly how to use a tool.. The schema is everything:

JSON
{
  "name": "query_client_records",
  "description": "Query the client database by ID, email, or name. Returns contact info, account status, and dispute history. Use for SPECIFIC client lookups, NOT for aggregate stats.",
  "input_schema": {

    "properties": {
      "client_id": {"type": "string"},

    }
  }
}

The description is prompt engineering. Bad description = wrong tool picked = wrong answer.

Key Concept

Tools are apps on the agent's phone

A tool is just something the agent can use, like search, a data store, or email. Clear tool rules help the agent pick the right one.

✧ Analogy

Tools are like apps on your phone. The name + icon + description determines whether you open Maps or Uber when you need to get somewhere. The agent reads the descriptions and picks.

  2. A function that runs Python code

› Feynman Check
Why is the tool description more important than the system prompt?
Answer

The system prompt shapes personality and approach. Tool descriptions shape capability and choice. A well-described tool in a poorly-prompted agent still gets used correctly. A badly-described tool gets misused no matter how good the system prompt. The description is how the agent knows what the tool is for.

Chapter 05

Retrieval patterns — the three shapes.

25 minRouter, Sequential, Parallel
◈ Learning Objective

Know the three fundamental patterns and when to use each.

Three patterns for shaping retrieval — pick based on whether sub-queries depend on each other.

Pattern A: Router (single-shot with intelligence)

Agent picks ONE tool per query. No loop.

Use when: questions cleanly belong to one data source.
Example: "What's our refund policy?" → vector search. "How many clients signed up yesterday?" → SQL.

Pattern B: Sequential (the classic loop)

Agent uses tools one at a time, chaining results.

Use when: later queries depend on earlier results.
Example: "Look up client Mo's disputes, then check if any are past the 30-day FCRA window."

Pattern C: Parallel (fan-out)

Agent fires multiple tools at once, merges results.

Use when: queries are independent and you want speed.
Example: "Get our Q3 financials, recent customer reviews, and competitor pricing."

Key Concept

Why there are three, not one

Some questions need one search. Some need steps in order. Some need many searches at once. That is why there are different retrieval patterns.

✧ Analogy

Router = a receptionist directing you to the right office.
Sequential = a detective following leads one by one.
Parallel = a research team splitting up to cover ground faster.

⟡ Exercise

For each question, pick the pattern:

  1. "What's the weather in Tokyo?" → Router
  2. "Find similar clients to Sarah, then check their dispute success rates." → Sequential
  3. "Summarize today's news, stock prices, and my calendar." → Parallel
› Feynman Check
When would parallel fail where sequential succeeds?
Answer

When Query B depends on the output of Query A. Parallel can't know Query A's result at dispatch time. Sequential waits. Example: "find Sarah's account id, then look up her disputes" — you can't look up disputes without knowing the id first, so parallel fails.

Chapter 06

Building one from scratch.

45 minCode with Claude's tool-use API
◈ Learning Objective

Write a working agentic RAG system using Claude's tool-use API.

The minimum viable agent

Python
import anthropic

client = anthropic.Anthropic()

# 1. Define tools
tools = [
    {
        "name": "search_docs",
        "description": "Semantic search over company documentation. Use for policy, how-to, and conceptual questions.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Natural language search query"}
            },
            "required": ["query"]
        }
    },
    {
        "name": "query_database",
        "description": "Run SQL against the customer database. Use for specific records, filters, and aggregations. Tables: customers, orders, tickets.",
        "input_schema": {
            "type": "object",
            "properties": {
                "sql": {"type": "string"}
            },
            "required": ["sql"]
        }
    }
]

# 2. Implement tools
def execute_tool(name, inputs):
    if name == "search_docs":
        return vector_db.search(inputs["query"], top_k=5)
    elif name == "query_database":
        return db.execute(inputs["sql"])

# 3. The agent loop
def agentic_rag(user_question, max_iterations=10):
    messages = [{"role": "user", "content": user_question}]

    for i in range(max_iterations):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=4096,
            system="You are a research assistant. Use tools to gather information before answering. Cite sources.",
            tools=tools,
            messages=messages
        )

        messages.append({"role": "assistant", "content": response.content})

        # Agent wants to use tools
        if response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": str(result)
                    })
            messages.append({"role": "user", "content": tool_results})
        else:
            # Agent is done
            return response.content[0].text

    return "Max iterations reached."

That's a production agentic RAG system at its core. Everything else is optimization.

Key Concept

What stop_reason is telling you

This flag tells your code what the model wants next. tool_use means run a tool. end_turn means the model is done for now.

⟡ Exercise

Build this for a real use case:

  • Pick a domain you know (credit disputes, payroll, inventory)
  • Define 2 tools (one semantic, one structured)
  • Test with 5 questions and watch the loop
› Feynman Check
What happens on iteration 3 if the agent asks for a tool that doesn't exist?
Answer

The API returns an error (or your execute_tool dispatcher returns a KeyError). Your code should catch it and feed a tool_result back explaining the tool doesn't exist — the agent will read the error, self-correct, and try a different approach on the next iteration. This is why structured error messages from tools matter: the agent treats them as feedback.

Chapter 07

Production hardening.

30 minSeven deadly problems
◈ Learning Objective

Know every failure mode and how to prevent it.

The seven deadly problems

△ 1. Infinite loops

Symptom: agent keeps calling tools, never answers.
Fix: max_iterations cap (5–10), force final answer on limit.

△ 2. Context explosion

Symptom: token count grows each iteration, LLM degrades.
Fix: summarize tool results, truncate old observations, use sliding window.

△ 3. Wrong tool selection

Symptom: agent SQL-queries for conceptual questions, vector-searches for filters.
Fix: explicit "use for X, NOT for Y" in tool descriptions.

△ 4. Hallucinated tool arguments

Symptom: agent invents column names, client IDs, or parameters.
Fix: put schema in system prompt; validate inputs; return clear errors.

△ 5. Cost explosion

Symptom: each query costs dollars, not cents.
Fix: cache tool results; use Haiku for routing decisions; cap iterations aggressively.

△ 6. Stale retrievals

Symptom: agent returns outdated info.
Fix: re-embed changed docs; add timestamps to retrieved chunks; force freshness checks.

△ 7. Poor groundingLike showing the real receipt when someone asks what you bought. The answer stays tied to real proof.

Symptom: agent makes up facts despite having tools.
Fix: system prompt requires citations; post-check answers against tool outputs.

Key Concept

Why agents hallucinate even with tools

Tools do not stop mistakes by magic. The agent must use real tool results and base its answer on proof. If it does not, it can still make things up.

The production checklist

  • Max iterations capped (5–10)
  • All tool calls logged with inputs, outputs, latency, tokens
  • Tool errors return structured messages the agent can parse
  • Cost per query tracked, alerts on anomalies
  • Evaluation suite of 20–50 gold-standard Q&A pairs
  • Fallback behavior when tools fail
  • Rate limiting on expensive tools
  • Schema validation on tool inputs
  • Timeout on each tool call (don't hang forever)
  • Structured output format for the final answer
› Feynman Check
You ship v1. Users complain answers are "sometimes wrong." What's your debugging order?
Answer
  1. Check logs: did the agent even call tools?
  2. Check tool outputs: did tools return correct data?
  3. Check tool descriptions: is the agent picking the right tool?
  4. Check system prompt: is the agent being told to cite/verify?
  5. Check eval suite: can you reproduce the failure?

Chapter 08

Advanced patterns.

20 minDemo → product
◈ Learning Objective

Know the patterns that separate hobbyist demos from production systems.

Pattern 1: Query decompositionLike turning 'clean the house' into room-by-room jobs. One big question becomes smaller ones.

Break complex queries into sub-queries before retrieving.

Flow
"Compare CA and TX client dispute success in Q3"
    ↓
Sub-query 1: CA clients, Q3 disputes, success rate
Sub-query 2: TX clients, Q3 disputes, success rate
    ↓
Synthesize comparison

Pattern 2: Multi-agent / specialist agents

An orchestrator delegates to specialized sub-agents.

Topology
Orchestrator
    ├── Legal Agent (knows FCRA, FDCPA)
    ├── Data Agent (SQL, analytics)
    └── Writing Agent (formats responses)

Why: each sub-agent has focused tools, tighter prompts, and its own context.

Pattern 3: Self-correction / reflection

After generating an answer, the agent critiques itself (a reflection passLike rereading a text before you send it. The system checks and improves its first try.).

Flow
Draft answer → "Does this answer cite sources?" → If no, retrieve more → Revise

Pattern 4: HyDELike sketching what you hope to find before searching a messy closet. That rough draft helps the search. (Hypothetical Document Embeddings)

Before retrieving, the agent writes a hypothetical answer, then searches for documents similar to that hypothetical.

Why it works: answers are semantically closer to docs than questions are.

Key Concept

Why HyDE feels like cheating

HyDE first writes a fake answer, then searches for real pages that look like it. This can find better matches than the raw question alone.

Pattern 5: Adaptive retrievalLike deciding whether you need GPS before driving. The system first checks if a lookup is needed.

The agent learns when to retrieve vs answer from memory.

Decision
"What is 2+2?" → answer directly (no retrieval)
"Who won the 2025 World Series?" → retrieve
"What did I ask you yesterday?" → check memory, not retrieval
✧ Analogy

Demo RAG is a karaoke singer with one mic. Production RAG is a live band — decomposition splits the song into parts, specialist agents are each instrument, reflection is the recording engineer listening back, HyDE is the lead singer rehearsing the melody before the take, adaptive retrieval is knowing when to let silence speak.

› Feynman Check
When would you use multi-agent over a single agent with many tools?
Answer

When (a) you need isolated context windows per domain, (b) different domains need different system prompts, or (c) sub-tasks can run in parallel. Single agent is simpler; multi-agent scales better and handles cross-domain work where one system prompt can't optimize for everything.

Evaluation & iteration.

◈ Learning Objective

The evaluation loop

  1. Build a dataset of 20–50 questions with known-good answers
  2. Run your system on all of them
  3. LLM-as-judge scores each answer (accuracy, groundedness)
  4. Human spot-checks 10% of the LLM judgments
  5. Fix the worst failures → re-run → compare metrics
  6. Never ship without metrics trending up
Key Concept

Why "it feels better" isn't enough

Gut feeling is not enough. Test the old version and the new version with the same questions, then compare the results.

› Feynman Check
Your retrieval precision is 90% but answer accuracy is 60%. What's wrong?
Answer

You're retrieving good stuff but the LLM isn't using it properly. Look at: (1) system prompt — are you telling it to ground in retrievals? (2) answer format — is it citing? (3) whether retrieved chunks actually contain the answer or just look similar to the question. Relevant chunks don't always carry the answer — sometimes they carry related but different content.

NextFinal Feynman Test — Prove it→

Final Feynman test.

Three questions. Pick the best answer. Wrong answers teach you something too.

An engineer tells you her company has 10 million product descriptions in a vector DB, and her team needs to answer: "Which SKUs from the Q3 launch are outperforming their initial forecast by more than 20%?" What's the first thing you tell her?

The question is a filter and aggregation problem ("outperforming by more than 20%"), not a semantic similarity problem. Vector search returns semantically similar chunks — it doesn't evaluate numeric thresholds. The right architecture is an agent with a SQL tool (for the numeric filter) that can optionally chain to vector search (for descriptive context). Chapter 2's core lesson: structured filters need structured queries. Chapter 5: this is a sequential pattern — SQL first, then optionally enrich with semantic retrieval.

Your agent keeps looping — it calls search_docs, reads the result, then calls search_docs again with a slightly reworded query, and again, burning iterations without answering. What's the most likely root cause?

This is a wrong tool selection problem (Chapter 7, deadly problem #3) manifesting as infinite loop symptoms (#1). Raising max_iterations is a band-aid. The real fix is tool descriptions that tell the agent what each tool is for and when to stop using it — often including an explicit "if search_docs doesn't find the answer in 2 tries, try [other tool] or return that you don't know." Tool descriptions aren't documentation — they're prompt engineering for the agent's decision-making.

In one sentence: what does the stop_reason field in Claude's response control in an agentic RAG loop?

stop_reason is the control flag of the agent loop. After every model response, your orchestrator checks it: "tool_use" means "I want to call a tool — here's which one" so you execute the tool and feed the result back via another turn. "end_turn" means "I'm done — this is my final answer." Without reading this field, you can't run an agentic loop; with it, the whole pattern from Chapter 6 works. It's not a token limit, not a safety filter — it's a state machine signal.
